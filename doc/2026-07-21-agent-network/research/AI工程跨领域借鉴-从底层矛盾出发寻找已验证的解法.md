# AI 工程跨领域借鉴：从底层矛盾出发，寻找已验证的解法

> **写在前面**：这不是一篇"AI 工程像 XXX"的类比文章。类比是廉价的——任何两个领域都能找到表面相似点。本文要做的是：识别 AI 工程面临的**底层矛盾**，然后在其他已成熟的领域中找到**解决同类矛盾的已验证方案**，最后给出一个可落地的思考框架。如果你也在思考"AI 工程化到底该往哪个方向走"，希望这些跨学科的视角能提供一些结构化的思路。

---

## 一、历史回顾：软件工程借鉴土木工程的得与失

### 1.1 借了什么

1968 年，NATO 软件工程会议首次提出"软件工程"概念。当时的核心焦虑是：软件项目频繁失控——延期、超预算、质量低下。解决思路很自然：向最成熟的工程学科取经。

软件工程从土木工程借鉴了三样东西：

| 借鉴内容 | 土木工程原型 | 软件工程应用 |
|---------|------------|------------|
| **阶段式过程管理** | 勘察→设计→施工→验收 | 需求→设计→编码→测试（瀑布模型） |
| **模块化与标准化** | 预制构件、建筑规范 | 设计模式、编码标准、接口协议 |
| **文档驱动** | 施工图纸是法律文件 | 需求规格说明书、设计文档 |

这些借鉴在当时确实有效——至少把"写程序"从"手工艺"提升为了"工程活动"。

### 1.2 代价是什么

但这个借鉴有一个隐含假设：**变更成本曲线相似**。

在土木工程中，变更成本随阶段指数级上升——设计阶段改一条线是免费的，施工阶段拆一面墙代价巨大。瀑布模型正是基于这个假设设计的：用前期的重度设计来避免后期的昂贵变更。

但软件的变更成本曲线完全不同。改一行代码的成本不随项目阶段剧烈变化（有版本控制、有自动化测试、有 CI/CD）。**软件工程花了整整 20 年**——从 1970 年代到 1990 年代——才从这个不匹配的假设中挣脱出来，走向迭代和敏捷。

### 1.3 教训

> **借鉴的前提不是"看起来像"，而是"底层矛盾是否同构"。**

软件和建筑的生产过程表面相似（都有设计→实现→验收），但底层矛盾完全不同：
- 建筑的核心矛盾：物理约束 + 不可逆变更 → 需要前重后轻
- 软件的核心矛盾：需求模糊 + 廉价变更 → 需要快速反馈迭代

把"前重后轻"的方案搬到"需要快速迭代"的领域，就会水土不服。

**这个教训是本文的出发点。**

## 二、追问：AI 工程的底层矛盾是什么？

### 2.1 AI 工程不是软件工程的简单延伸

很多人把 AI 工程（特指用 LLM/Agent 辅助研发的工程实践）当作软件工程的自然扩展。但它面临的核心矛盾是全新的：

| 矛盾维度 | 传统软件工程 | AI Agent 工程 |
|---------|------------|-------------|
| **工具确定性** | 编译器：相同输入 → 相同输出 | LLM：相同 prompt → 不同输出 |
| **规格可描述性** | 需求可以写成精确 spec | 很多意图难以完全形式化 |
| **错误可复现性** | Bug 稳定复现，可以写回归测试 | 同一 prompt 下次可能正常 |
| **质量验证手段** | 断言、类型系统、确定性测试 | 输出正确性往往需要人类判断 |
| **复杂度来源** | 代码量 × 模块交互 | 上下文组合爆炸 × 模型概率性 |

**AI 工程的核心矛盾一句话概括：**

> 你的工程需求是**确定性**的（交付可靠的软件），但你的核心工具是**概率性**的（LLM 的每次输出都有随机性）。

这是一个前所未有的张力——传统软件工程从未面对过"工具本身不确定"的问题。

### 2.2 三个子矛盾

这个核心矛盾进一步分解为三个具体挑战：

1. **概率输出 vs 确定性需求**：如何从不稳定的 AI 输出中获得可靠的工程产物？
2. **有限窗口 vs 无限上下文**：如何在 Token 约束下让 AI 拥有足够的"知识"来做正确决策？
3. **个人经验 vs 团队标准化**：如何让一个人摸索出的最佳实践变成整个团队的能力？

### 2.3 关键转折：向哪里寻找解法？

面对这些矛盾，一个自然的想法是：有没有"更成熟的上游学科"可以借鉴？就像当年软件工程借鉴土木工程一样？

答案是：**没有。**

AI Agent 工程已经处于工程学科的前沿，上面没有一个"AI 的土木工程"等着你去取经。

但这不意味着无处借鉴。正确的策略是：

> **不向上找更成熟的母体学科，而向旁边找——找那些面临相同底层矛盾、且已发展出成熟解法的领域。**

```
1968: 土木工程 ──"过程管理"──→ 软件工程
          （新工程向老工程学管理方法）
              ↓
        但底层矛盾不同 → 20年弯路 → 敏捷
              
2025: AI 工程面临全新矛盾
          ↓
      不存在"AI 的土木工程"
          ↓
      向旁边找"解决相同底层矛盾"的领域：
      
      ┌─────────────┐   ┌────────────┐   ┌────────────┐
      │  质量工程    │   │  控制论    │   │  人因工程   │
      │(驾驭概率性) │   │(反馈闭环) │   │(人机分工)  │
      └─────────────┘   └────────────┘   └────────────┘
```

## 三、方向一：质量工程——如何驾驭概率性输出

### 3.1 为什么同构

制造业面临的核心问题：**单个产品有随机波动（加工公差），但整体良品率必须可控。**

这和 AI 工程完全同构：
- 单次 LLM 调用有概率性波动
- 但最终交付物的质量必须可靠

制造业用了 80 年（从 1920 年代 Shewhart 的质量控制理论到 Six Sigma）来解决这个问题。核心方法论已经高度成熟。

### 3.2 可迁移的具体方案

#### 方案 A：统计过程控制（SPC）思维

制造业的洞察：**不是检查每一个产品的对错，而是监控整个过程是否"在控"。**

一个加工过程如果"在控"（in-control），即使单个产品有波动，整体良品率也是可预测的。只有当过程"失控"（out-of-control）时，才需要人工干预。

**迁移到 AI 工程：**

| SPC 概念 | AI 工程对应 |
|---------|-----------|
| 过程能力（Cpk） | 一套 prompt + context 配置的"稳定产出能力" |
| 控制图（Control Chart） | 监控 AI 输出质量的趋势，而非纠结单次结果 |
| 特殊原因变异 | Agent 输出突然偏离——需要归因（是 prompt 退化？context 过载？模型更新？） |
| 普通原因变异 | LLM 固有的概率性——接受它，通过系统设计来包容 |

**可落地的框架：**

```
AI 输出质量 = f(Prompt设计, Context质量, 模型能力, 任务复杂度)

当输出质量波动时：
  → 先判断是"普通原因"还是"特殊原因"
  → 普通原因：接受波动，通过门禁和审查来兜底
  → 特殊原因：定位根因（5-Why），修正系统（更新 prompt/context/流程）
```

#### 方案 B：防错设计（Poka-Yoke）

丰田生产系统的核心理念：**不是靠检查来发现错误，而是通过设计让错误不可能发生。**

经典例子：USB 接口设计成只能正确方向插入，而不是贴一个"请注意方向"的标签。

**迁移到 AI 工程：**

| 防错层级 | 制造业示例 | AI 工程对应 |
|---------|----------|-----------|
| **消除**（让错误不可能） | 只有正确方向能插入 | 分支保护规则阻止直接 push master |
| **预防**（在发生前阻止） | 超重自动停机 | 门禁检查（阶段产出不完整则不允许进入下一阶段） |
| **检测**（发生后立即报警） | 零件计数器 | 代码审查 Checker（security/concurrency/error） |

**关键区别：** 传统代码审查是"检测型"防错——错误已经发生了，审查员去找到它。而更高级的防错是"预防型"——让 AI 在开始编码之前就有正确的设计约束和上下文，从源头减少犯错可能。

**可落地的设计原则：**

```
防错设计三层模型：

Layer 1 - 消除层（系统级）
  → 分支隔离：每个需求独立分支，物理隔离错误传播
  → 目录规范：通过结构约束让 AI 知道"文件该放哪"

Layer 2 - 预防层（流程级）
  → 强制门禁：关键阶段必须通过审查才能推进
  → 上下文注入：让 AI 开始工作时就拥有正确的规范和约束
  → 计划确认：任务开始前先写计划并获得人确认

Layer 3 - 检测层（执行级）
  → 多维度 Checker：设计一致性/安全性/并发安全/错误处理/复杂度
  → 追溯链验证：需求→设计→代码→测试 全链路可追溯
  → 过程记录：process.txt 作为"黑匣子"，异常时可回溯
```

### 3.3 小结

质量工程给 AI 工程的核心启示：

1. **接受概率性，通过系统设计来包容它**——不要试图让每次 AI 输出都完美，而是设计一个系统使得即使单次输出有波动，整体交付质量依然可控
2. **从"终检"升级到"过程控制"**——不只是最后做一次 code review，而是在关键节点设置门禁，监控过程是否在控
3. **防错优于检错**——与其事后发现 AI 写了有安全漏洞的代码，不如事前就把安全规范注入 context

## 四、方向二：控制论——反馈闭环设计

### 4.1 为什么同构

控制论（Cybernetics）诞生于 1948 年 Norbert Wiener 的同名著作，核心解决的问题是：**如何让一个不完全可预测的系统保持稳定运行？**

飞行器、自动驾驶、工业机器人——这些系统都有一个共同特点：它们不可能100%按预期运行（有风干扰、有路况变化、有负载波动），但必须保持在可接受范围内。

解法的核心思想：**不要试图一次做对，而是通过快速感知偏差 + 快速修正来逼近目标。**

AI Agent 工程面临的问题高度类似：
- AI 的输出不可能100%符合预期
- 但通过反馈循环可以逼近正确结果
- 关键不是消除偏差，而是**缩短偏差被发现和修正的周期**

### 4.2 PID 控制的工程启示

PID 控制器是工业界最广泛使用的反馈控制方案，包含三个分量：

| PID 分量 | 控制论含义 | AI 工程映射 |
|---------|----------|-----------|
| **P（比例）** | 对**当前偏差**的响应强度 | 当次 AI 输出与设计预期的差距 → 即时纠偏（code review 发现问题 → 修复） |
| **I（积分）** | 对**历史偏差累积**的消除 | 同类任务反复出现的偏差模式 → 根因修正（更新 context 规范，消除系统性偏差） |
| **D（微分）** | 对**偏差变化趋势**的预判 | 偏差是在收敛还是发散？ → 决策是否需要更换策略（回退阶段/换方案） |

**核心洞察：** 只有 P（即时纠偏）是不够的——如果你只做 code review 发现问题→修复，但不做 I（把经验写回规范）和 D（判断方向是否正确），系统就会反复在同一个地方出问题。

### 4.3 可落地的闭环设计框架

```
┌─────────────────────────────────────────────────────────────┐
│                    AI 工程闭环控制系统                         │
│                                                             │
│   ┌─────────┐     ┌─────────┐     ┌─────────┐             │
│   │ 设定目标 │────→│ AI 执行  │────→│ 输出产物 │             │
│   │(需求/设计)│     │(Agent)  │     │(代码/文档)│             │
│   └─────────┘     └─────────┘     └────┬────┘             │
│        ↑                                │                   │
│        │              反馈修正           │                   │
│        │                                ↓                   │
│   ┌────┴────────────────────────────────────────┐           │
│   │              三层反馈回路                      │           │
│   │                                              │           │
│   │  内环（P - 即时修正，周期：分钟级）            │           │
│   │  → code review 发现问题 → 当场修复             │           │
│   │  → 门禁不通过 → 补充产出后重试                 │           │
│   │                                              │           │
│   │  中环（I - 模式修正，周期：需求级）            │           │
│   │  → 同类问题反复出现 → 更新 context 规范         │           │
│   │  → 经验沉淀到知识库 → 下次 AI 自动避坑          │           │
│   │                                              │           │
│   │  外环（D - 方向修正，周期：项目级）            │           │
│   │  → 评估整体趋势 → 流程是否需要调整              │           │
│   │  → 工具/方法论是否需要升级                      │           │
│   └──────────────────────────────────────────────┘           │
└─────────────────────────────────────────────────────────────┘
```

**三环嵌套的关键设计点：**

1. **内环必须快**：发现偏差到修正的周期越短越好。这就是为什么需要自动化门禁和 Checker——不能等人工 review 才发现问题
2. **中环必须沉淀**：每次修正如果只停留在"当次修复"，就相当于只有 P 没有 I，系统不会进步
3. **外环必须有勇气**：当整体趋势表明当前方案不可行时，要有"回退阶段"甚至"推翻重来"的能力

### 4.4 小结

控制论给 AI 工程的核心启示：

1. **不追求一次做对，追求快速收敛**——设计系统的目标不是"让 AI 第一次就输出完美结果"，而是"让偏差被快速发现和修正"
2. **多层反馈回路**——即时修正（P）+ 模式修正（I）+ 方向修正（D）缺一不可
3. **记录是修正的前提**——没有 process.txt 这样的"过程记录"，就无法做趋势分析和根因归因

## 五、方向三：人因工程——人和 AI 如何分工

### 5.1 为什么同构

人因工程（Human Factors Engineering）是一个有 50 年历史的严格学科，专门研究：**人和自动化系统如何分配任务，以达到整体效能最优？**

这不是天马行空的联想——这是一个解决"人机协作边界"问题最成熟的学术领域。Aviation、Nuclear、Healthcare 三大高风险行业的人机协作规范都源自这个学科。

AI 工程面临的分工问题和它高度一致：
- 哪些任务交给 AI，哪些留给人？
- AI 做决策时，人应该审批、监督、还是完全放手？
- 随着 AI 能力提升，边界如何动态调整？

### 5.2 Sheridan-Verplank 自动化分级模型

1978 年 Sheridan 和 Verplank 提出了 10 级自动化分级：

| 等级 | 描述 | AI 工程对应 |
|:---:|------|-----------|
| 1 | 人完全手工决策，系统无参与 | 传统编程，不用 AI |
| 2 | 系统提供备选方案 | AI 提供多个方案让人选 |
| 3 | 系统推荐一个方案，人决定是否采纳 | AI 写好代码，人 review 后决定是否用 |
| 4 | 系统执行推荐方案，除非人否决 | AI 执行，人有否决权（有时间窗口） |
| 5 | 系统执行，但事前告知人 | AI 自动执行并通知人 |
| 6 | 系统执行，仅在人主动询问时告知 | AI 后台运行，人想看时才看 |
| 7 | 系统执行，仅异常时通知人 | AI 自主运行，出错时才上报 |
| 8-10 | 系统完全自主，人无参与 | 全自动 AI 流水线 |

**关键洞察：不同任务/阶段应该对应不同的自动化等级。**

### 5.3 可落地的分级框架

将 AI 工程中的不同活动按自动化等级分类：

```
┌─────────────────────────────────────────────────────────────┐
│           AI 工程自动化等级分配框架                           │
├───────────────┬──────┬─────────────────────────────────────┤
│ 活动类型       │ 等级 │ 人的角色                             │
├───────────────┼──────┼─────────────────────────────────────┤
│ 架构决策       │  2   │ AI 列方案，人做最终选择               │
│ 需求澄清       │  3   │ AI 结构化后人确认                    │
│ 代码生成       │  4   │ AI 生成，人有否决权（code review）    │
│ 单元测试生成   │  5   │ AI 自动生成并执行，结果通知人         │
│ 代码格式化     │  7   │ AI 自动执行，仅异常时通知             │
│ 进度记录       │  7   │ AI 自动更新 process.txt              │
│ 门禁检查       │  5   │ AI 自动检查，结果需人确认             │
│ 知识沉淀建议   │  3   │ AI 建议沉淀什么，人决定是否采纳       │
└───────────────┴──────┴─────────────────────────────────────┘
```

### 5.4 自动化自满（Automation Complacency）的风险

人因工程的另一个核心发现：**当自动化系统运行良好时，人会逐渐降低监控频率，直到完全依赖系统——此时系统一旦出错，人已经失去了发现和纠正的能力。**

这个风险在 AI 工程中极其现实：
- 如果门禁检查每次都通过，人会开始"自动点确认"而不真正看内容
- 如果 AI code review 每次都说 PASS，人会降低自己审查代码的意愿
- 当有一天 AI 产出了一个隐蔽的逻辑错误，而人已经不再认真看——损害就发生了

**应对策略：**

1. **强制人工介入点**：在关键决策节点（架构选择、对外接口设计），不允许纯自动通过
2. **随机深度审查**：即使 AI 审查通过，也定期抽取进行人工深度 review
3. **能力保持训练**：团队成员定期不依赖 AI 独立完成任务，保持手感
4. **透明化 AI 不确定性**：AI 输出应明确标注"高确信/中等确信/猜测"，让人知道什么该多看一眼

### 5.5 小结

人因工程给 AI 工程的核心启示：

1. **不同任务给不同自动化等级**——不是所有事都让 AI 全权做，也不是所有事都要人审批
2. **自动化等级应该随信任积累而渐进提升**——新任务类型从 Level 3 开始，验证稳定后升到 Level 5
3. **防范自动化自满**——设计系统时就要考虑"如何让人保持在回路中"

---

## 六、整合：一个可落地的思考框架

### 6.1 三个方向的协同关系

三个借鉴方向不是独立的，而是解决 AI 工程核心矛盾的三个层面：

```
AI 工程核心矛盾：概率性工具 × 确定性需求

解法层面一（质量工程）：如何从概率输出中获得可靠产物？
  → 答案：过程控制 + 防错设计 + 接受波动设计兜底

解法层面二（控制论）：如何设计让系统持续改进的反馈机制？
  → 答案：三层闭环（即时/模式/方向）+ 过程记录 + 知识沉淀

解法层面三（人因工程）：人和 AI 各做什么？边界在哪？
  → 答案：分级授权 + 渐进信任 + 防范自满
```

### 6.2 框架总结：AI 工程的"驾驭模型"

综合三个方向，提出一个结构化的思考框架：

```
┌─────────────────────────────────────────────────────────┐
│              AI 工程"驾驭模型"（Harness Model）           │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ 层一：防错设计（来自质量工程）                       │  │
│  │   • 消除层：结构约束让错误不可能发生                 │  │
│  │   • 预防层：门禁让错误无法传播到下游                 │  │
│  │   • 检测层：多维 Checker 快速发现已发生的偏差        │  │
│  └───────────────────────────────────────────────────┘  │
│                          ↕                              │
│  ┌───────────────────────────────────────────────────┐  │
│  │ 层二：反馈闭环（来自控制论）                         │  │
│  │   • 内环（分钟级）：发现→修复                       │  │
│  │   • 中环（需求级）：模式识别→规范更新→知识复利       │  │
│  │   • 外环（项目级）：趋势评估→方向调整                │  │
│  └───────────────────────────────────────────────────┘  │
│                          ↕                              │
│  ┌───────────────────────────────────────────────────┐  │
│  │ 层三：分级授权（来自人因工程）                       │  │
│  │   • 按任务类型分配自动化等级（2-7级）               │  │
│  │   • 随信任积累渐进提升自主度                        │  │
│  │   • 强制人工介入点防范自动化自满                     │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  贯穿原则：                                             │
│  1. 接受概率性，通过系统设计驾驭它（而非消灭它）         │
│  2. 不追求一次做对，追求快速收敛                         │
│  3. 每次实践产生复利（知识沉淀 → 下次更好）             │
│  4. 人的价值在于方向判断和异常处理，不在于重复执行       │
└─────────────────────────────────────────────────────────┘
```

### 6.3 实践映射示例

以一个典型的 AI 辅助需求开发流程为例，三层如何协同：

| 环节 | 防错设计（质量工程） | 反馈闭环（控制论） | 分级授权（人因工程） |
|------|-------------------|------------------|-------------------|
| **需求定义** | 模板约束（必填项） | 需求评审反馈→修订 | Level 3（AI 结构化，人确认） |
| **技术预研** | 强制门禁（不通过不进下阶段） | 风险评估→方案调整 | Level 2（AI 列方案，人选择） |
| **详细设计** | 追溯链检查（需求↔设计一致性） | 设计评审→迭代完善 | Level 3（AI 撰写，人审核） |
| **编码实现** | 多 Checker 并行审查 | review 发现→修复→规范更新 | Level 4（AI 编码，人有否决权） |
| **测试验收** | 自动化测试执行 | 测试失败→定位→修复→回归 | Level 5（AI 执行，通知人） |
| **知识沉淀** | — | 经验→context/→下次复利 | Level 3（AI 建议，人决定） |

## 七、什么不能借鉴？——避免过度类比的陷阱

诚实地说，以下"看似可以借鉴"的方向需要警惕：

### 7.1 敏捷方法论不能直接照搬

敏捷的核心假设：需求会变，所以要小批量快速交付、频繁收集反馈。

但 AI 工程中的"需求变化"和传统软件不同：
- 不是"客户改了主意"
- 而是"AI 理解和人意图之间的偏差"——这更像通信中的噪声，而非信号本身的变化

所以：Sprint 的"时间盒"概念不一定适用；更适合的是控制论中的"偏差收敛"模型。

### 7.2 微服务架构思维不能直接搬到多 Agent

微服务的核心优势是独立部署、独立扩缩。但多 Agent 系统中：
- Agent 之间的"通信成本"远高于微服务（每次通信都消耗大量 Token）
- 上下文在 Agent 间传递会大幅衰减
- 过细的拆分会导致"每个 Agent 都不够聪明"

教训：Agent 的拆分粒度应该由"任务自包含性"决定，而不是由"职责单一"决定。

### 7.3 航空 Checklist 不能只看形式

很多人看到"检查清单"就联想到航空业的 Checklist。但航空 Checklist 有效的前提是：
- 飞行员经过大量训练，理解每一项的含义
- 环境是标准化的（同一机型、同一程序）
- 违反检查单的后果极其严重（生命安全）

在 AI 工程中，如果门禁变成"走过场"——每次都点"确认"而不真正审查——就失去了意义。

**正确的借鉴方式：** 不是照搬 Checklist 的形式，而是借鉴其背后的"系统化防错"思想，并设计机制防止流于形式（如随机深度审查）。

## 八、结语：工程的本质是驾驭不确定性

回到开头的问题：AI 工程还能借鉴什么？

答案不是找一个"更成熟的 AI 土木工程"来照搬。而是：

1. **识别你面临的底层矛盾**（概率性 × 确定性、有限窗口 × 无限上下文、个人 × 团队）
2. **找到在其他领域已经解决过相同矛盾的方案**（质量工程、控制论、人因工程）
3. **验证迁移的前提是否成立**（底层矛盾同构，而非表面相似）
4. **在实践中验证和调整**（不存在"搬过来就能用"的银弹）

一切工程学科的本质都是同一件事：**在不确定性中创造可靠性。**

土木工程驾驭的是材料强度的不确定性。软件工程驾驭的是需求变化的不确定性。AI 工程驾驭的是模型输出的不确定性。

工具不同，但工程思维的内核是统一的。这大概就是"跨领域借鉴"的真正价值——不在于找到外衣相似的模仿对象，而在于发现那些跨越学科界限的、经过时间验证的**工程原理**。
