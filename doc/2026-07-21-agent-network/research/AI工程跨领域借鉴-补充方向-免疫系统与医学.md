## 方向 A（深化）：免疫系统 → AI 工程的"自适应防御体系"

### A.1 免疫系统的分层防御机制

人体免疫系统是自然界最精密的"自适应防御系统"——它不需要预先知道所有病原体长什么样，就能在遇到新威胁时自动产生针对性的防御，并记住这次经验用于未来。

```
人体免疫系统三层防御：

Layer 1: 先天免疫（Innate Immunity）
  → 物理屏障（皮肤/黏膜）：快速、非特异性、不学习
  → 对应 AI 工程：分支保护、目录规范、基础 lint
  → 特点：不管什么威胁一律挡住，零学习成本

Layer 2: 适应性免疫（Adaptive Immunity）
  → 特异性识别（T细胞/B细胞）：慢启动、精准、产生记忆
  → 对应 AI 工程：从具体失败中学习特定防御规则
  → 特点：精准定向，第一次慢，第二次秒级响应

Layer 3: 免疫记忆（Immunological Memory）
  → 记忆细胞长期存活，再次遇到相同病原时秒级响应
  → 对应 AI 工程：知识沉淀到 context/，全团队共享
  → 特点：一人踩坑，全团队免疫
```

### A.2 两种落地策略

#### 策略 1：暂存区 + 人工审核（胸腺选择模型）

免疫学中，T 细胞在释放到体内之前必须经过"胸腺选择"——正选择确保它能识别抗原，负选择确保它不攻击自身。只有两关都通过的 T 细胞才被释放。

```
AI 操作产生的"规则"同样需要"选择"过程：

  AI 发现模式
    → 生成候选规则（"抗体草案"）
    → 写入 staging_rules/（暂存区/胸腺）
    → 每次执行时 AI 可读取暂存区规则
    → 人定期审核：
        • 正选择：这个规则确实能防住问题吗？→ 升级到 context/
        • 负选择：这个规则会误伤正常代码吗？→ 丢弃或修正
```

**暂存区文件格式：**

```yaml
# staging_rules/RULE-2026-0509-001.yaml
id: RULE-2026-0509-001
created: 2026-05-09 14:30
source: code-review-report#EIV-191-F003
confidence: 0.85  # AI 自评置信度
status: staging   # staging / active / deprecated / rejected

pattern:
  description: "并发场景下直接使用 map 而非 sync.Map"
  detection: "在 goroutine 内使用非并发安全的 map 读写"

rule:
  content: "并发场景下读写 map 必须使用 sync.Map 或加锁保护"
  evidence_count: 3  # 被触发过的次数
  last_triggered: 2026-05-09

review_history: []  # 审核记录
```

**审核后的状态流转：**
```
staging → active（确认有效）→ 沉淀到 context/
staging → rejected（误报/不适用）→ 记录原因，后续不再触发
staging → modified（需要修正）→ 更新 rule 内容后重新观察
```

#### 策略 2：全自动 + Event Log + 可回退

对于低风险操作（如更新 INDEX、补充标签），可以采用更激进的策略——先自动生效，但保留完整操作日志，出问题时可回退。

```
Event Sourcing 模式：

  AI 每次对知识库的操作 → 写入不可变事件日志
  → 当前状态 = 顺序重放所有事件的结果
  → 任意时刻可回退到历史状态

事件日志格式：
---
event_id: EVT-2026-0509-001
timestamp: 2026-05-09T14:30:00+08:00
actor: AI/auto-extract
type: knowledge_added | knowledge_modified | knowledge_deprecated | rule_activated
target_file: context/project/dos/experience/concurrency-pitfalls.md
change_summary: "新增：goroutine 中使用 map 的并发安全经验"
trigger: code-review-report#EIV-191-F003
rollback_command: "git revert <commit-hash>"  # 如何撤销
---
```

**回退机制设计：**
```
用户："最近 AI 自动加的那条并发规则好像有问题"
  → 查 event log：按时间/类型/触发来源过滤
  → 找到对应事件：EVT-2026-0509-001
  → 执行回退：git revert 或手动删除
  → 记录回退原因（未来同类规则的参考）
```

## 方向 M：医学体系 → AI 工程的"诊疗系统"

### M1：现代医学的"循证体系"

#### M1.1 证据分级（Levels of Evidence）→ 知识可信度管理

现代医学最重要的方法论贡献：**不是所有"经验"都同等可信，需要按证据强度分级。**

```
医学证据金字塔（Oxford CEBM）：
  Level 1: 系统综述 / Meta 分析（多个 RCT 的汇总）
  Level 2: 随机对照实验（RCT）
  Level 3: 队列研究（前瞻性观察）
  Level 4: 病例对照研究（回顾性分析）
  Level 5: 专家意见 / 机理推断

映射到 AI 工程的知识可信度分级：

  Level 1 - 多需求验证 + 量化数据
    "sync.Map 在我们并发场景下比 mutex 快 30%（EIV-191, EIV-210, EIV-338 三个需求验证）"
    → AI 可高置信度引用，无需人确认
    
  Level 2 - 单需求完整验证
    "EIV-191 用 Redis 会话管理方案，测试全部通过，性能达标"
    → AI 可引用，标注"单次验证"
    
  Level 3 - Code Review 观察到的模式
    "连续 3 次 review 都发现 goroutine 中未处理 error"
    → AI 可引用作为风险提示，建议人关注
    
  Level 4 - 合理推断但未实证
    "这种写法理论上有内存泄漏风险（基于 Go 语言规范推断）"
    → AI 引用时必须标注 [推断，未实证]
    
  Level 5 - AI 自动提取的模式（未经人审核）
    → 仅作为参考，不可作为决策依据
    → 对应免疫系统的"暂存区"
```

**落地方式：** 在 context/ 文件的元数据中增加 evidence_level 字段：

```yaml
---
title: 并发安全最佳实践
evidence_level: 2  # 单需求验证
verified_by: [EIV-191]
verified_date: 2026-04-26
---
```

#### M1.2 临床路径 + 变异管理 → 标准化流程 + 合理偏离

临床路径（Clinical Pathway）的精髓不只是"标准化"，而是**同时定义了标准路径和偏离机制**：

```
标准路径（Happy Path）：
  八阶段流程：1→2→3→4→5→6→7→8

变异机制（Variance）：
  当实际情况不符合标准路径前提时，允许偏离，但必须：
  ① 记录偏离原因
  ② 评估偏离风险
  ③ 记录替代路径
  ④ 事后分析：是个例还是说明标准路径需要更新？

示例：
  标准：所有需求必须经过"阶段3技术预研"
  变异条件：bugfix 类需求，改动范围 < 50 行，有明确复现步骤
  变异决策：跳过阶段3、4，直接从阶段2进入阶段6
  记录：process.txt 中标注 [流程变异：跳过3/4，原因：简单bugfix]
  
  积累变异数据后的分析：
  → 如果 80% 的 bugfix 都跳过了阶段3/4
  → 说明应该为 bugfix 定义一条正式的精简路径
  → 更新标准流程，新增"bugfix 快速通道"
```

#### M1.3 MDT 会诊制度 → 多 Agent 分歧解决

```
当前 AME 模式：
  6 个 Checker 各自出报告 → code-review-report 汇总 → 人看汇总
  
问题：Checker 意见冲突时怎么办？
  例：design-checker 说"实现符合设计"
      complexity-checker 说"这个方法太复杂，应该拆分"
      → 遵循设计还是优化复杂度？

MDT 会诊升级：
  ① 意见一致时 → 直接汇总，正常流程
  ② 意见冲突时 → 触发"会诊"：
     • 明确列出分歧点和每个 Checker 的依据
     • 分析冲突本质（是约束冲突？还是优先级不同？）
     • 给出裁决建议（基于当前需求的特性）
     • 由"主诊"（人或主对话 AI）做最终裁决
     • 裁决及理由写入记录
  ③ 裁决积累后 → 提取模式：
     "当设计意图和代码复杂度冲突时，优先遵循设计（除非复杂度超过阈值X）"
     → 沉淀为规范，减少未来同类会诊
```

### M2：中医的"系统论方法"

#### M2.1 辨证论治 → 需求"辨证分型"

中医不按"病名"治疗，而是按"证型"——同一种病在不同体质/阶段可能需要完全不同的治法。

```
AI 工程"辨证分型"框架：

┌──────────┬─────────────────────────────────────────────────────────┐
│ 证型      │ 特征 & 策略                                            │
├──────────┼─────────────────────────────────────────────────────────┤
│ 表证      │ 改动浅层、范围小、有现成模式可循                         │
│（浅层）    │ 策略：精简流程（2→6→7→8），快速交付                    │
│           │ 例：修改配置项、修复明确的 bug                           │
├──────────┼─────────────────────────────────────────────────────────┤
│ 里证      │ 涉及架构变更、跨服务、核心链路改动                       │
│（深层）    │ 策略：完整流程（1→8），充分设计评审                      │
│           │ 例：新增服务模块、重构核心流程                           │
├──────────┼─────────────────────────────────────────────────────────┤
│ 虚证      │ 缺乏领域知识、没有参考代码、技术栈陌生                   │
│（能力不足）│ 策略：先"补虚"——加大调研、收集更多 context、找内部专家    │
│           │ 例：接入全新第三方系统、使用从未用过的技术                 │
├──────────┼─────────────────────────────────────────────────────────┤
│ 实证      │ 依赖阻塞、需求冲突、技术债务阻碍                         │
│（阻塞）    │ 策略：先"疏通"——解决阻塞后再推进                       │
│           │ 例：依赖其他团队的接口未就绪、历史代码质量太差需先重构     │
├──────────┼─────────────────────────────────────────────────────────┤
│ 寒热错杂  │ 部分清晰部分模糊、部分有经验部分是新领域                  │
│（混合型）  │ 策略：分而治之——清晰部分走快速通道，模糊部分走完整流程    │
│           │ 例：在熟悉的服务中接入陌生的外部 API                      │
└──────────┴─────────────────────────────────────────────────────────┘

辨证依据（四诊合参）：
  望：看需求描述的清晰度和完整度
  闻：听用户对紧急程度和风险的描述
  问：追问技术栈熟悉度、是否有参考实现、依赖是否就绪
  切：查看代码仓库现状、现有测试覆盖率、技术债务量
```

#### M2.2 治未病 → 预防性干预

```
"上工治未病，中工治欲病，下工治已病"

AI 工程的三层预防：

上工（阶段前预防）：
  → 需求阶段就预判实现风险
  → 基于历史"判例"，自动推送相关经验
  → 例：检测到需求涉及"并发"关键词 → 自动注入并发安全 checklist

中工（阶段中预警）：
  → 编码过程中实时检测潜在问题
  → 不等 code review 才发现
  → 例：AI 生成代码时自动对照设计文档，发现偏离立即提示

下工（阶段后修复）：
  → 当前模式：code review 发现问题 → 修复
  → 代价最大，已经"病了"才治
```

#### M2.3 君臣佐使 → Agent 协作的优先级编排

```
中药配伍"君臣佐使"原则：
  君：针对主病/主证，起主要治疗作用
  臣：辅助君药加强疗效，或治疗兼证
  佐：制约君臣之烈性，或反佐以防格拒
  使：引导药力到达病所，或调和诸药

映射到 Code Review 多 Checker 编排：

  默认配伍（通用需求）：
    君：design-checker（核心：实现是否符合设计意图）
    臣：error-checker + security-checker（重要：直接影响质量底线）
    佐：complexity-checker（制约：防止为了正确性过度复杂）
    使：auxiliary-checker（引导：确保改动范围正确，不漏不溢）

  安全敏感需求的配伍调整：
    君：security-checker（升为主力）
    臣：design-checker + error-checker
    佐：concurrency-checker（安全场景并发风险更高）
    使：auxiliary-checker

  性能优化需求的配伍调整：
    君：complexity-checker（核心：性能和复杂度的平衡）
    臣：design-checker
    佐：security-checker（制约：优化不能引入安全问题）
    使：auxiliary-checker

实际效果：
  • 报告按"君→臣→佐→使"优先级排序
  • "君"级问题必须修复才能通过
  • "佐"级问题如果和"君"冲突，优先保"君"
  • 不同需求类型自动调整"配伍方案"
```

#### M2.4 整体观 → 全局健康诊断

```
中医"整体观"：不只看局部病灶，看全身系统的平衡状态

AI 工程的"全局诊断"（当前缺失的能力）：

定期（如每周/每迭代）运行一次"团队健康诊断"：

┌─────────────────────────────────────────────────┐
│ 诊断维度         │ 检查内容                       │
├─────────────────┼────────────────────────────────┤
│ 知识覆盖均衡度   │ context/ 各领域文档覆盖度       │
│                 │ 哪些领域是"知识盲区"？           │
├─────────────────┼────────────────────────────────┤
│ 规则有效性       │ staging_rules 中哪些规则一直     │
│                 │ 未被触发（可能是过时的）？        │
├─────────────────┼────────────────────────────────┤
│ 流程偏离率       │ 多少需求走了完整流程 vs 精简流程 │
│                 │ 偏离的原因是否说明流程需要调整？  │
├─────────────────┼────────────────────────────────┤
│ 跨需求冲突       │ 并行进行的需求之间有无潜在冲突？ │
│                 │ 修改同一文件？依赖同一接口？      │
├─────────────────┼────────────────────────────────┤
│ 知识衰减度       │ 多少 context/ 文件超过90天未更新 │
│                 │ 是"稳定"还是"遗忘"？             │
├─────────────────┼────────────────────────────────┤
│ 复利效率         │ 新需求是否越来越多引用已有知识？ │
│                 │ 知识库的"投入产出比"趋势         │
└─────────────────┴────────────────────────────────┘

输出："团队健康报告"
  → 像体检报告一样，标注绿/黄/红
  → 绿：健康（知识充足、流程顺畅）
  → 黄：注意（某领域知识薄弱、规则可能过时）
  → 红：警告（有冲突风险、知识严重缺失）
```
